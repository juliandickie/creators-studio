# v3.6.0 Implementation Plan — Vertex AI Backend

**Working title:** Add a Vertex AI backend path to `video_generate.py` so the plugin can call the full VEO 3.1 capability surface (Lite, GA `-001` IDs, image-to-video, and Scene Extension v2) using API-key auth, then un-gate everything that v3.5.0 marked Vertex-only.

---

## Why this is happening

Real-API verification during the v3.5.0 release cycle (2026-04-10) discovered that the Gemini API surface (`generativelanguage.googleapis.com/v1beta`) the plugin had been using **does not serve** any of the following:

- `veo-3.1-lite-generate-001` (HTTP 404)
- `veo-3.1-generate-001` and `veo-3.1-fast-generate-001` GA IDs (HTTP 404)
- `veo-3.0-generate-001` Legacy (HTTP 404)
- `--video-input` Scene Extension v2 (HTTP 400 "inlineData isn't supported by this model")
- `--first-frame` / `--last-frame` / `--reference-image` image-to-video (same HTTP 400)

v3.5.0 shipped with these features documented and gated behind clear runtime errors pointing at this v3.6.0 work. The `--quality-tier draft` flag was remapped from Lite ($0.05/sec) to Fast ($0.15/sec) as a stopgap. The full draft-then-final cost story (8× savings vs Standard) is blocked until this plan ships.

## Why this is now feasible without OAuth

A second round of empirical testing on 2026-04-11 (with the user's bound-to-service-account API key) discovered that **`predictLongRunning` IS reachable via API-key auth on the standard Vertex AI endpoint**, even though Google's own Express Mode docs only list `generateContent`/`streamGenerateContent`/`countTokens`.

**Docs-level confirmation** (2026-04-11 follow-up research): the Google REST reference page for [`endpoints.predict`](https://docs.cloud.google.com/vertex-ai/generative-ai/docs/reference/rest/v1/projects.locations.endpoints/predict) **omits the "Authorization scopes" section** that OAuth-only methods list. Per Google's own documentation convention, omitting that section indicates API key authentication is officially supported. The `predictLongRunning` method (the async variant of the same `PredictionService.Predict` RPC) is governed by the same rule. Combined with our three successful empirical tests, this is both docs-level and runtime-level confirmation of the auth approach.

**The Express Mode "3-method allowlist" applies to anonymous API keys only.** Bound-to-service-account API keys (like the one we're using, format `AQ.Ab8...`, created through the Vertex Express signup flow) access the full `PredictionService` method list including `predictLongRunning`. This explains the apparent contradiction between the Express Mode docs and our successful empirical tests.

**Canonical protobuf type names** (from the REST reference):
- Request instance type: `VideoGenerationModelInstance`
- Request parameters type: `VideoGenerationModelParams`
- Response wrapper type: `PredictResponse`
- Veo-specific response payload: `cloud.ai.large_models.vision.GenerateVideoResponse` (visible in `response.@type` field of our empirical results)

**Optional field from the schema we're not using:** `labels` is a `map<string, string>` in the request body, used for tagging requests for billing/observability grouping. Not in scope for v3.6.0 — worth a v3.6.1 enhancement if we want to tag generations by user/session/feature for cost attribution.

The empirical request that worked:

```
POST https://us-central1-aiplatform.googleapis.com/v1/projects/{PROJECT}/locations/us-central1/publishers/google/models/veo-3.1-lite-generate-001:predictLongRunning?key={API_KEY}
Content-Type: application/json

{
  "instances": [{
    "prompt": "...",
    "image": {"bytesBase64Encoded": "...", "mimeType": "image/png"}   // optional
  }],
  "parameters": {
    "sampleCount": 1,
    "durationSeconds": 4,
    "aspectRatio": "16:9",
    "resolution": "720p"
  }
}
```

Polling:

```
POST https://us-central1-aiplatform.googleapis.com/v1/projects/{PROJECT}/locations/us-central1/publishers/google/models/veo-3.1-lite-generate-001:fetchPredictOperation?key={API_KEY}
Content-Type: application/json

{"operationName": "projects/.../operations/..."}
```

Done response shape:

```json
{
  "done": true,
  "response": {
    "@type": "type.googleapis.com/cloud.ai.large_models.vision.GenerateVideoResponse",
    "raiMediaFilteredCount": 0,
    "videos": [
      {"bytesBase64Encoded": "...", "mimeType": "video/mp4"}
    ]
  }
}
```

Verified end-to-end on 2026-04-11 with three real generations (~$0.75 total spend):

- ✅ Lite text-to-video (4s, 720p, 16:9): 1.69 MB MP4, 30s wall clock — `/tmp/v360-verify/lite-test.mp4`
- ✅ Lite image-to-video (4s, 720p, 16:9, with first-frame seed): 4.71 MB MP4, 40s wall clock — `/tmp/v360-verify/lite-i2v-test.mp4`
- ✅ Lite Scene Extension v2 (7s, 720p, 16:9, inline base64 source video): 4.52 MB MP4, 50s wall clock — `/tmp/v360-verify/scene-ext-v2-verified.mp4`

**Critical Scene Extension v2 findings:**

1. **The `feature: video_extension` code path accepts ONLY `durationSeconds: 7`.** Values 4, 6, and 8 (which work for text/image modes) are rejected with `"Unsupported output video duration X seconds, supported durations are [7] for feature video_extension"`. The existing `HOP_DURATION = 7` constant in `video_extend.py` is correct; the existing `GENERATE_DURATION = 8` constant is **wrong** when `--method video` is used and will fail every time against Vertex. Commit 3 must fix this.

2. **Inline base64 video input works** via `instances[0].video.bytesBase64Encoded` + `mimeType: "video/mp4"`, even though Google's only-documented path is `gcsUri`. The API silently stages the bytes through GCS internally. Our 1.69 MB source clip worked fine.

3. **First Scene Extension v2 call on a cold Vertex project returns a transient error:** `"Service agents are being provisioned (https://cloud.google.com/vertex-ai/docs/general/access-control#service-agents). Service agents are needed to read the Cloud Storage file provided. So please try again in a few minutes."` This is Google auto-provisioning the Vertex AI Service Agent role for the project's system-managed service account. Text-to-video and image-to-video do NOT hit this error because they don't touch GCS internally. Resolution: sleep ~60-90s and retry. Commit 2 needs to detect this error specifically and either retry automatically or surface a clear retry-later message.

4. **Per-operation wall clock** for Lite on Vertex: text-to-video ~30s, image-to-video ~40s, Scene Extension v2 ~50s. The existing 10s poll interval and 300s max-wait defaults in `video_generate.py` are correct.

| Resolution string is `"4K"` (uppercase) | **Vertex docs use lowercase `"4k"`.** Whether the parser is case-insensitive is unverified. | [generate-videos-from-text reference](https://docs.cloud.google.com/vertex-ai/generative-ai/docs/video/generate-videos-from-text) literal value: `"4k"` |

These corrections need to land in commit 4's doc updates.

## Doc-validated additions to the request body shape (2026-04-11 follow-up read)

Re-read of the [generate-videos-from-text reference page](https://docs.cloud.google.com/vertex-ai/generative-ai/docs/video/generate-videos-from-text) confirmed the full `parameters` object schema:

```json
{
  "instances": [{"prompt": "..."}],
  "parameters": {
    "storageUri":      "gs://bucket/output/",   // OPTIONAL — write directly to GCS
    "sampleCount":     1,                       // 1-4
    "aspectRatio":     "16:9",                  // "16:9" | "9:16" only
    "negativePrompt":  "...",                   // OPTIONAL — confirmed field name
    "personGeneration":"allow_adult",           // OPTIONAL — "allow_adult" | "disallow"
    "resolution":      "720p",                  // "720p" | "1080p" | "4k" (lowercase)
    "seed":            42                       // OPTIONAL — 0..4294967295
  }
}
```

**Note:** `durationSeconds` is documented in the duration constraint text ("4, 6, or 8") but is **not** shown in the parameters JSON example on the reference page. Our 2026-04-11 smoke test used `durationSeconds: 4` and it worked end-to-end, so the field name is confirmed empirically — Google's docs are just incomplete on this one. Track as a doc bug to file with Google later, but the field name is correct.

**`generateAudio` and `enhancePrompt`** are NOT documented Vertex parameters. They were Gemini API extensions that the plugin never used anyway. Skip them.

**`personGeneration`** is new — the plugin currently doesn't set it. Default behavior on Vertex is undocumented but our smoke tests worked without it (the lake-and-boat prompt didn't include people, so we may have dodged the question by accident). **Defer to v3.6.1** as a new optional flag — out of scope for v3.6.0 since the existing default works for everything we test.

**Valid values for `personGeneration`:** from the [official Google Veo 3 notebook](https://github.com/GoogleCloudPlatform/generative-ai/blob/main/vision/getting-started/veo3_video_generation.ipynb) the values are **`"allow_adult"` or `"dont_allow"`** — NOT `"disallow"` as the `generate-videos-from-text` reference page claimed. The notebook is more authoritative because it's actually running code.

**`generateAudio` IS a supported parameter** (boolean) per the same notebook, despite the earlier doc fetch incorrectly saying it didn't exist. Not used in v3.6.0 (plugin defaults to audio-on for all generations), but flagged in the parameters section of `veo-models.md` docs.

**`google-cloud-aiplatform` Python SDK is being deprecated in May 2026**, replaced by `google-genai`. Irrelevant for the plugin (we use raw HTTP, not either SDK), but worth a one-line mention in `veo-models.md` so users following the official Google guides know which SDK to install if they want a non-plugin path.

**google-genai SDK issue #2079 — v1 vs v1beta1 routing bug.** The google-genai Python SDK has a known unresolved bug ([#2079](https://github.com/googleapis/python-genai/issues/2079)) where Veo 3.1 GA model calls are routed to `v1beta1` instead of `v1`, producing the same `RESOURCE_PROJECT_INVALID` error I saw in my first probe. Irrelevant for this plugin (we use raw HTTP and can fix our URL explicitly), but the `_vertex_backend.py` helper module should **hard-code `/v1/` in the URL template** and add a regression test that fails if anyone ever sneaks `v1beta1` in.

**`sampleCount` vs `number_of_videos`** — the official notebook uses `number_of_videos` at the SDK level, which the SDK likely translates to either `sampleCount` or `numberOfVideos` at the REST level. My smoke test used `sampleCount: 1` and it worked; the request was accepted and produced a real video. For v3.6.0 we can stick with `sampleCount` (empirically proven). If a future v3.6.1 adds `--num-videos` support for 1–4 variations, we should verify whether `sampleCount: N > 1` actually returns N videos or gets aliased to `numberOfVideos: N` under the hood — this is an unknown that doesn't block v3.6.0.

## Execution plan: 5 commits + verification + release

### Commit 1: New `_vertex_backend.py` helper module (no behavior change)

**File:** `skills/video/scripts/_vertex_backend.py` (NEW, ~280 lines)

The Vertex path is structurally similar enough to the existing Gemini API path that we could inline it into `video_generate.py`. But the request-shape translator + response-shape translator are big enough that they deserve their own module for testability and to keep `video_generate.py` under 600 lines.

This module is a **pure data-translation layer with no global state** so commit 1 ships it standalone, with no caller yet, and we can test it in isolation:

1. **Constants**
   - `VERTEX_ENDPOINT_TEMPLATE = "https://{region}-aiplatform.googleapis.com/v1/projects/{project}/locations/{region}/publishers/google/models/{model}:{method}?key={key}"`
   - `VERTEX_GLOBAL_ENDPOINT_TEMPLATE = "https://aiplatform.googleapis.com/v1/projects/{project}/locations/global/publishers/google/models/{model}:{method}?key={key}"` (for `location=global`)
   - `VERTEX_DEFAULT_LOCATION = "us-central1"`

2. **`build_vertex_url(model, method, project, location, api_key)`**
   - Returns the full POST URL with the API key in the query string
   - `method` is one of `"predictLongRunning"` or `"fetchPredictOperation"`
   - Branches on `location == "global"` to use the global endpoint template

3. **`build_vertex_request_body(prompt, *, duration, aspect_ratio, resolution, image_path=None, video_input_path=None, negative_prompt=None, seed=None, sample_count=1)`**
   - Returns the dict to JSON-serialize for `:predictLongRunning`
   - Reads image file → base64 if `image_path` is set, attaches as `instances[0].image.bytesBase64Encoded` + `mimeType` (PNG/JPEG/WebP detection from extension)
   - Reads video file → base64 if `video_input_path` is set, attaches as `instances[0].video.bytesBase64Encoded` + `mimeType: "video/mp4"` (this MAY fail with "video too large" for Scene Extension v2 — gate this in commit 4 with the same `gcsUri` warning v3.5.0 has)
   - **Normalizes `resolution` to lowercase** before sending — Vertex docs use `"4k"` (lowercase); the plugin's existing `"4K"` (uppercase) is converted at the boundary. Apply `.lower()` only for `"4K" → "4k"`; `"720p"` and `"1080p"` are already lowercase.
   - **Validates `aspect_ratio` is in `{"16:9", "9:16"}`** — `"1:1"` is rejected with a clear message ("Vertex AI VEO 3.1 only supports 16:9 or 9:16; 1:1 was deprecated"). This catches plan.json files written under the v3.5.0 wrong-doc claim that Lite supports 1:1.
   - Maps `aspect_ratio` and `resolution` (normalized) straight through into `parameters`
   - Maps `duration` to `parameters.durationSeconds` (note: `durationSeconds` not `duration` — empirically confirmed even though Google's docs example is missing this field)
   - `negative_prompt` becomes `parameters.negativePrompt` (field name confirmed by [generate-videos-from-text reference](https://docs.cloud.google.com/vertex-ai/generative-ai/docs/video/generate-videos-from-text))
   - `seed` becomes `parameters.seed` (field name confirmed; valid range 0..4294967295)
   - **Does not set `personGeneration`** — defer to v3.6.1 as a new optional flag

4. **`parse_vertex_submit_response(response_dict)`**
   - Returns the `operation_name` string from the submit response
   - Raises `RuntimeError` with a helpful message on failure

5. **`parse_vertex_poll_response(response_dict)`**
   - Returns one of `("running", None)`, `("done", video_bytes)`, or `("error", error_message)`
   - `video_bytes` is the decoded base64 from `response.videos[0].bytesBase64Encoded`
   - Handles the case of multiple videos (sample_count > 1) by returning a list of bytes when len > 1
   - Handles `raiMediaFilteredCount > 0` by returning an error indicating the content was RAI-filtered

6. **`vertex_diagnose()` (CLI entry point)**
   - When `_vertex_backend.py` is run directly (`python3 _vertex_backend.py`), it reads creds from `~/.banana/config.json`, prints the resolved URL pattern, and pings the global endpoint with a tiny `gemini-2.5-flash-lite:generateContent` call to verify the auth works
   - Useful for debugging "is my Vertex setup working" without burning VEO budget
   - Cost: $0 (Gemini text gen is cheap and free at low volume)

### Commit 2: Wire the Vertex backend into `video_generate.py`

**File:** `skills/video/scripts/video_generate.py` (+150 lines, -30 lines)

1. **New `--backend {gemini-api,vertex-ai,auto}` flag**
   - Default: `auto`
   - `auto` routes:
     - Models in `MODELS_VERTEX_ONLY` → `vertex-ai`
     - `--video-input` set → `vertex-ai`
     - `--first-frame` / `--last-frame` / `--reference-image` set → `vertex-ai` (image-to-video is Vertex-only)
     - Otherwise → `gemini-api` (preserves the working text-to-video path on the Gemini API for users who haven't configured Vertex)
   - Add `--vertex-project` and `--vertex-location` flags (override config)

2. **Refactor `_submit_operation()` and `_poll_operation()`**
   - Both functions get a `backend` parameter
   - Branch on backend at the top of each function:
     - `gemini-api`: existing code path (no behavior change)
     - `vertex-ai`: call `_vertex_backend.build_vertex_url()` + `build_vertex_request_body()` + POST + parse with `parse_vertex_submit_response()`. Polling uses `:fetchPredictOperation` POST instead of GET on the operation name.

3. **Refactor `_save_video()`**
   - Branch on backend
   - `gemini-api`: existing GET-with-API-key download path
   - `vertex-ai`: bytes already in hand from `parse_vertex_poll_response()`, just write to disk and return path. **No 48-hour expiry warning needed for Vertex.**

4. **Backend-aware error messages**
   - When `--backend vertex-ai` is requested but `vertex_api_key`/`vertex_project_id` is missing from config, error with a specific setup pointer (link to the GCP console + the v3.6.0 ROADMAP entry for the setup steps)
   - **Service agent provisioning detection:** when polling returns `error.code == 9` (FAILED_PRECONDITION) with a message containing `"Service agents are being provisioned"`, emit a stderr warning `"First Scene Extension v2 call on this Vertex project. Service agents auto-provision in ~60-90s. Retrying in 90s..."` and retry ONCE after a 90-second sleep. If the retry also fails with the same error, raise with a clear user-facing message pointing at the [Vertex AI service agents doc](https://cloud.google.com/vertex-ai/docs/general/access-control#service-agents).
   - **Scene Extension v2 duration gate:** if `--video-input` is set and `--duration` is not 7 (or not explicitly set and the default is 8), override to 7 with a `_progress` warning `"Scene Extension v2 requires durationSeconds=7; overriding from X"`. This matches the plugin's existing auto-downgrade-to-720p pattern for Scene Extension v2 resolution.

5. **Remove the gates added in v3.5.0 reality-check commit (`7d9da65`)**
   - Delete the `MODELS_VERTEX_ONLY` preflight check (or rather, repurpose it as the `auto` backend selector)
   - Delete the `VIDEO_INPUT_VERTEX_ONLY` gate (the Vertex backend handles `--video-input`)
   - Re-enable image-to-video flag handling in the Vertex code path

6. **Update `download_expires_at`** logic
   - Only emit `download_expires_at` and the 48h stderr warning when `backend == "gemini-api"`. Vertex returns inline base64 with no expiry.

### Commit 3: Un-gate Lite, GA IDs, image-to-video; remap draft tier; Lite duration fix

**Files:** `skills/video/scripts/video_generate.py`, `skills/video/scripts/video_sequence.py`, `skills/video/scripts/video_extend.py`

1. **`video_generate.py`: fix the Lite duration constraint**
   - Replace `LITE_DURATIONS = set(range(5, 61))` with `LITE_DURATIONS = STANDARD_DURATIONS = {4, 6, 8}` (remove the special-case)
   - Remove `_valid_durations()` model-aware lookup if Lite is no longer special — this becomes a one-line constraint
   - This is a v3.5.0 doc-correction, but the code change happens here because v3.5.0 still gated Lite from being callable, so the wrong constraint was harmless

2. **`video_generate.py`: remove the Lite 1:1 special-case (no probe needed)**
   - The 2026-04-11 doc re-read confirmed Vertex VEO 3.1 only supports `16:9` and `9:16`. The v3.5.0 1:1 claim was wrong on two counts (it was also wrong about the duration range).
   - Delete `LITE_RATIOS = {"16:9", "9:16", "1:1"}` and replace with the same `STANDARD_RATIOS = {"16:9", "9:16"}` for all models. No model-aware ratio lookup needed anymore.
   - The request body builder also enforces this at the Vertex boundary as a defense-in-depth check.
   - **No real-API probe needed** — saved $0.20 of verification budget vs the original plan.

3. **`video_sequence.py`: re-point `--quality-tier draft` to Lite**
   - Revert `QUALITY_TIER_MODELS["draft"]` from `veo-3.1-fast-generate-preview` back to `veo-3.1-lite-generate-001`
   - Update help text: drop the "currently maps to Fast" note

4. **`video_extend.py`: re-enable Scene Extension v2 as the default**
   - Flip `--method` default from `keyframe` back to `video`
   - Update help text accordingly
   - **Critical fix:** Change `GENERATE_DURATION = 8` to `GENERATE_DURATION_VIDEO_EXT = 7` for the scene-extension path. Scene Extension v2 rejects any duration except 7. The existing `HOP_DURATION = 7` constant was already correct — only `GENERATE_DURATION` (what gets passed to the VEO call) was wrong.
   - Keep `HOP_DURATION = 7` unchanged (matches the actual effective second-gain per hop, which happens to equal the new generation duration for the scene-extension path).
   - **Caveat:** Scene Extension v2 with inline base64 video has been verified working at 1.69 MB. The existing `_read_video_base64()` 15 MB cap is the right guardrail — if a user hits it, the error should point at "upload to GCS and use `gcsUri` instead" which is a v3.6.1 enhancement, not a v3.6.0 blocker.
   - **Service agent cold-start handling:** `video_generate.py` (via the Vertex backend) should detect the transient "Service agents are being provisioned" error pattern specifically and emit a `_progress` event with `"status": "service_agent_provisioning", "retry_after_seconds": 90` so callers can decide whether to retry. Commit 2 is where this detection lands; commit 3 just makes sure `video_extend.py` propagates the error clearly to the user.

5. **`video_generate.py`: update the model validation tip messages**
   - Re-point all "tip: use veo-3.1-fast-generate-preview" messages back to "tip: use veo-3.1-lite-generate-001 for cheap drafts"

## Decisions baked into this plan that you should approve before I start

1. **API-key auth, not OAuth.** Confirmed empirically. The plan does not pull in `google-auth` or shell out to `openssl` or `gcloud`. Pure stdlib + your existing config file pattern. ✅
2. **Inline base64 video bytes on the Vertex path, no GCS upload.** Confirmed working for 4-second Lite clips up to 4.7 MB. May fail for larger Scene Extension v2 inputs above 15 MB — that's a known TODO and would land in v3.6.1 with a GCS upload helper if it surfaces.
3. **`--backend auto` is the default**, not an explicit user opt-in. The Gemini API path is preserved for text-to-video on Standard/Fast preview IDs (the v3.4.x-compatible default), and the Vertex path activates whenever a Vertex-only model or feature is requested. This means **no breaking changes** for existing v3.5.0 users — their text-to-video calls keep working through the same backend, and their previously-gated calls now succeed transparently.
4. **`--quality-tier draft` reverts to Lite**, not Fast. This is the whole point.
5. **Scene Extension v2 ships as the new default for `video_extend.py`** with the inline base64 path. If it fails for clips > ~10 MB, users get a clear error and we add GCS upload in v3.6.1.
6. **Lite duration is {4, 6, 8}** — the v3.5.0 5-60 range was wrong and is being corrected here.

## Known risks for the fresh implementation session

- **Commit 2 is the most dangerous** because it changes how `_submit_operation()` and `_poll_operation()` route. The existing Gemini API code path must remain byte-identical for `--backend gemini-api`. Add a regression test that submits a tiny text-only request via the explicit `--backend gemini-api` flag and confirms the payload matches the v3.5.0 shape exactly.

- **The empirical request body might be incomplete for `negativePrompt` and `seed`.** I tested `prompt`, `image.bytesBase64Encoded`, `video.bytesBase64Encoded`, `sampleCount`, `durationSeconds`, `aspectRatio`, `resolution` empirically. The 2026-04-11 doc re-read confirmed `negativePrompt` and `seed` field names from the [generate-videos-from-text reference](https://docs.cloud.google.com/vertex-ai/generative-ai/docs/video/generate-videos-from-text). Risk reduced — but commit 2's smoke test should still exercise both to make sure they don't surface a "field present but unused" silent failure.

- ~~**The 1:1 aspect ratio claim for Lite is unverified.**~~ **RESOLVED 2026-04-11 doc re-read.** Vertex docs explicitly list only `16:9` and `9:16`. No probe needed; just delete the special-case code.

- ~~**Scene Extension v2 may not actually work on Vertex for Veo 3.1.**~~ **RESOLVED 2026-04-11 empirical probe.** Scene Extension v2 works end-to-end via inline base64 video input. Real 4.52 MB MP4 at `/tmp/v360-verify/scene-ext-v2-verified.mp4`.

- ~~**Scene Extension v2 may require GCS upload.**~~ **RESOLVED 2026-04-11 empirical probe.** Inline `bytesBase64Encoded` works for clips up to at least 1.69 MB source. Docs-only `gcsUri` path still exists as a fallback for larger clips, deferred to v3.6.1.

- **NEW: Service agent cold-start on fresh Vertex projects.** First Scene Extension v2 call returns `"Service agents are being provisioned"` with code 9. Transient — auto-resolves in ~60-90s. Commit 2 needs to detect this specific error and retry once automatically after 90s, OR surface a clear user-facing retry message. Text-to-video and image-to-video are NOT affected (they don't touch GCS internally).

- **NEW: Scene Extension v2 requires `durationSeconds=7`.** Other durations rejected with `"supported durations are [7] for feature video_extension"`. Commit 3 fixes `video_extend.py` `GENERATE_DURATION` and adds an auto-override in `video_generate.py` when `--video-input` is set.

- **Resolution case sensitivity is unverified.** The plugin uses `"4K"` (uppercase); Vertex docs show `"4k"` (lowercase). The request body builder normalizes to lowercase as a precaution. Commit 2's smoke test should explicitly try a `4K` resolution on Standard via `--backend vertex-ai` to confirm the normalization works (cost: $3.20 for one Standard 8s 4K clip — **OR** test Lite at 720p which is what we already ran; the 4K case never hits Lite anyway since Lite doesn't support 4K). **Decision:** skip the 4K probe in commit 2's verification; the normalization is cheap insurance and the 4K path is rarely used.

- **The bound API key may have a per-region quota** that surfaces only after the first few real generations. Watch for HTTP 429 in commit 5's verification. If it happens, document the quota path in `veo-models.md` and add retry logic in a follow-up.

- **`gemini-2.5-flash-lite` is spelled with hyphens in the user's curl example but the actual Gemini API model is `gemini-2.5-flash` without `-lite`.** Don't propagate the typo into the diagnose CLI in commit 1.

- **The plugin's `_load_api_key()` precedence is `CLI flag → env var → config`.** Make sure the new `_load_vertex_credentials()` follows the same precedence so users who set `VERTEX_API_KEY=...` in their shell can override the config file.

- **Don't store the API key as a literal in any code or commit.** It's already in `~/.banana/config.json` (gitignored). The only places it appears in source should be the documentation note that says "saved at ~/.banana/config.json" and the function that reads it.

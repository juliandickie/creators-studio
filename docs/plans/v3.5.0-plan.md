### Critical: GA model IDs vs preview IDs (affects commits 2, 4)

**Finding:** The second doc reveals that VEO 3.1 has GA (Generally Available) production model IDs ending in `-001`, not `-preview`:

| Tier | Preview ID (in commit 1) | GA ID (from second doc) |
|---|---|---|
| Standard | `veo-3.1-generate-preview` ✅ tested working | `veo-3.1-generate-001` - official GA |
| Fast | `veo-3.1-fast-generate-preview` ✅ passes validation | `veo-3.1-fast-generate-001` - official GA |
| Lite | `veo-3.1-lite-generate-001` ✅ (already the GA ID) | `veo-3.1-lite-generate-001` - same |
| Legacy | `veo-3.0-generate-001` - already GA | same |

**Decision needed in fresh session:**

Our commit 1 shipped the preview IDs for Standard and Fast (because those are what we had evidence worked, from our earlier successful VEO test). The GA IDs are what Google now officially recommends in their latest docs.

**Recommended approach: merge both namespaces.** Add the GA IDs alongside the preview IDs in `VALID_MODELS` so plans written at any time continue to work. Keep the default as `veo-3.1-generate-preview` for now (tested working) but add a doc note saying the GA IDs are the forward path. This is additive and zero-risk.

```python
VALID_MODELS = {
    # GA (official production IDs, recommended going forward)
    "veo-3.1-generate-001",
    "veo-3.1-fast-generate-001",
    "veo-3.1-lite-generate-001",
    "veo-3.0-generate-001",
    # Preview (legacy names, kept for backward compatibility with v3.4.x plans)
    "veo-3.1-generate-preview",
    "veo-3.1-fast-generate-preview",
}
```

When commit 2 updates `cost_tracker.py`, the PRICING dict should have entries for both ID families with the same rates. When commit 4 updates the docs, explicitly recommend GA IDs with the preview IDs marked as "accepted for backward compat."

**Alternative: switch defaults to GA and deprecate preview.** More aggressive, risks breaking workflows that hardcoded `veo-3.1-generate-preview`. Not recommended for v3.5.0 - save for v3.6.0 after GA IDs have been exercised in production.

### Token limit: 1,024 tokens max per prompt (affects commit 2, optional)

**Finding:** VEO 3.1 accepts prompts up to 1,024 tokens, English only. Our current `video_generate.py` has no length validation at all - a 5,000-token prompt would silently fail at the API layer with a confusing error.

**Optional new step for commit 2 (or commit 3):** Add a simple token-count pre-check in `video_generate.py`. Since we don't have a tokenizer dependency (stdlib only), use a heuristic: **roughly 1 token ≈ 4 characters for English prose.** So 1,024 tokens ≈ 4,096 characters. Conservative threshold: warn at 3,800 characters, error at 4,500 characters. Place the check right after `args = parser.parse_args()` so it fails fast.

```python
# Rough token estimate: ~4 chars per token for English prose. VEO 3.1 limit is 1,024 tokens.
PROMPT_WARN_CHARS = 3800   # ~950 tokens, warn
PROMPT_ERROR_CHARS = 4500  # ~1,125 tokens, hard reject
if len(args.prompt) > PROMPT_ERROR_CHARS:
    _error_exit(
        f"Prompt is {len(args.prompt)} characters (~{len(args.prompt)//4} tokens). "
        f"VEO 3.1 limit is 1,024 tokens (~4,096 characters). "
        f"Shorten the prompt or split into multiple shots."
    )
elif len(args.prompt) > PROMPT_WARN_CHARS:
    _progress({
        "warning": "prompt_approaching_limit",
        "chars": len(args.prompt),
        "estimated_tokens": len(args.prompt) // 4,
        "limit_tokens": 1024
    })
```

### 2-day video retention (affects commits 2, 4)

**Finding:** Generated videos persist on Google's servers for only 2 days. After that, the download URI expires. Our current `video_generate.py` downloads immediately after generation so we're not affected at runtime, but **we've been storing video URIs in manifests without knowing they expire.** This is an operational footgun for anyone who re-runs a stale plan.

**Change for commit 2:** Update the JSON output of `video_generate.py` to include a `download_expires_at` timestamp (current time + 48 hours) so downstream tools can warn users if they're trying to act on an expired URI. Add a warning printed to stderr after successful generation: `"Note: The source download URI expires in 48 hours. The MP4 has been saved to disk."`

**Change for commit 4 (docs):** Add a "Video retention" section to `skills/video/references/veo-models.md` explaining the 48-hour window, recommending `output_gcs_uri` for production workflows (not yet supported by our script but worth documenting for users who want that path), and warning that manifest JSONs with `uri` fields become stale.

### 4 videos per request (optional for v3.5.0)

**Finding:** VEO 3.1 supports `number_of_videos` parameter with values 1-4. Generating 4 variations in a single API call is cheaper than 4 sequential calls because the model amortizes the setup/denoise overhead. Our script hardcodes `sampleCount: 1`.

**Decision:** This is a NEW feature, not a gap fix. Adding it means a new `--num-videos` flag, handling multiple output files, and a new output JSON shape (array of paths instead of single path). That's real scope creep for v3.5.0.

**Recommendation: defer to v3.6.0** and add as a ROADMAP entry. The one-shot variation use case overlaps with the A/B testing feature in the banana skill, so it might belong there (e.g., `/banana ab-test` could internally generate 4 variations per prompt in a single API call). Flag for future design discussion.

### 4K is AI-upscaled from 1080p, not native (affects commit 4 docs only)

**Finding:** VEO's 4K output (3840×2160) is generated by AI-upscaling a 1080p base generation, not by natively rendering at 4K. This was added January 2026.

**Impact:** Pricing doesn't change - 4K is still $0.40/sec for Standard. But this explains why 4K is only available on Standard (the upscale quality is tuned for the flagship tier, not Lite/Fast).

**Change for commit 4:** Add a clarification in `veo-models.md`: "Note: 4K output is produced via AI-powered upscaling from a 1080p base generation. The generation cost is the same as 1080p; the upscale step is included. This explains why 4K is available only on Standard tier."

### Timestamp prompting within a single clip (affects commit 4 docs)

**Finding:** VEO 3.1 supports a timestamp syntax within the prompt text to direct multi-shot sequences WITHIN a single 8-second generation. Example:

```
[00:00-00:02] Medium shot from behind a young explorer pushing aside a jungle vine.
[00:02-00:04] Reverse shot of the explorer's face, expression filled with awe. SFX: rustle of leaves.
[00:04-00:06] Tracking shot following her hand over intricate carvings on a wall.
[00:06-00:08] Wide crane shot revealing the vast temple complex. SFX: swelling orchestral score.
```

**This is a completely new prompting technique we haven't documented.** It effectively lets you pack 2-4 micro-shots into one 8-second clip, which dramatically changes the shot planning math. A 30-second sequence could be 2 clips with 4 sub-shots each instead of 4 clips with 1 shot each - half the VEO calls, half the cost.

**Change for commit 4:** Add a new "Timestamp prompting" section to `skills/video/references/video-prompt-engineering.md` with the above example and these caveats:

- Works within the 4/6/8-second hard ceiling - cannot extend clip length
- Each sub-shot must make sense as a micro-cut (establishing → reaction → detail → wide)
- Audio cues can be placed at specific timestamps (SFX alignment)
- Total sub-shot duration must not exceed the clip duration
- Quality trade-off: tight sub-shots at high count can look rushed; 2-3 sub-shots per clip is the sweet spot

**Update the draft-then-final workflow section** in `video-sequences.md` to mention that timestamp prompting can reduce total clip count for sequences, potentially saving 40-50% on video generation costs for certain shot lists.

### Dialogue timing guidance (affects commit 4 docs)

**Finding:** "Keep dialogue short - roughly what can be spoken in one breath within an 8-second window. Too much dialogue causes unnaturally rapid speech; too little produces AI-generated gibberish filling the silence."

**Change for commit 4:** Add to `video-prompt-engineering.md` as a new tip under the audio section:

- 4-second clip: 3-5 words of dialogue max (e.g., `A woman says, "I'm ready."`)
- 6-second clip: 6-10 words (e.g., `He whispers, "Let's go before they see us."`)
- 8-second clip: 10-15 words (one full breath of natural speech)
- Don't leave dialogue completely absent in a clip with a visible speaker - VEO fills silence with gibberish murmuring

### Lens focal length guidance (affects commit 4 docs)

**Finding:** "Lenses control depth perception, not distance - use 16mm to expand space, 35mm for natural perspective, 85mm for background compression."

**Change for commit 4:** Add a "Lens focal length" section to `video-prompt-engineering.md` in the camera vocabulary area:

| Focal length | Effect | Use case |
|---|---|---|
| 16mm | Expands space, exaggerates depth | Establishing shots, environments, dramatic interiors |
| 24-35mm | Natural perspective, close to human vision | Standard coverage, documentary feel, medium shots |
| 50mm | Slight compression, flattering proportions | Portrait, conversation shots, product closeups |
| 85mm | Heavy background compression, creamy bokeh | Portrait beauty, isolating subject from environment |
| 100mm+ macro | Extreme close-up with very shallow DOF | Product details, textures, pour shots |

VEO responds accurately to specific focal lengths (e.g., "50mm f/1.2"), so use them.

### Character drift is a documented limitation (affects commit 4 docs)

**Finding:** "VEO treats each prompt as a fresh generation with no persistent character memory, producing 'character drift' - subtle or dramatic changes in facial features, clothing, and hairstyle between clips. Mitigations (reference images, verbatim character descriptions, image-to-video) help but don't fully solve this. Competitors Kling 2.6 and Seedance 2.0 currently handle multi-shot character consistency better."

**Change for commit 4:** Add a "Known limitations" section to `veo-models.md` that includes character drift, text rendering unreliability, the 8-second clip ceiling, and occasional silent-output failures. Be honest about the trade-offs so users pick VEO when its strengths match their use case.

Also update the "Scene bible anchors" tip in `video-prompt-engineering.md` (which we wrote earlier in this session) to explicitly frame it as a **mitigation** for character drift, not a complete solution.

### Regional restrictions (affects commit 4 docs only)

**Finding:** Image-to-video is restricted in EEA, Switzerland, and UK. `personGeneration` defaults to `allow_adult` only in those regions.

**Change for commit 4:** Add a "Regional availability" section to `veo-models.md` noting the restrictions. Not a code change (our script doesn't set `personGeneration` at all in commit 1, which is correct for text-to-video).

### Gemini + VEO workflow officially recommended (affects commit 4 docs)

**Finding:** "Google recommends a Gemini + VEO workflow for best results: use Gemini 2.5 Flash to generate reference frames, expand simple prompts into cinematic language, and create first/last frame images - then feed these into VEO 3.1."

**Change for commit 4:** This is exactly the pattern the coffee shop demo uses - `/banana generate` (Gemini) for storyboard frames, then `/video generate --first-frame/--last-frame` (VEO) for clips. Add a sentence to `video-sequences.md` noting that this workflow is Google's official recommendation, not just our opinion. Marketing/credibility win, not a code change.

### Text rendering unreliable (affects commit 4 docs)

**Finding:** "Text rendering remains VEO 3.1's most conspicuous weakness. Text on signs, shirts, and storefronts consistently renders as plausible-looking gibberish. Never rely on the model for readable text - add it in post-production."

**Change for commit 4:** Add to the "Known limitations" section of `veo-models.md` AND to the "Banned Keywords" style section of `video-prompt-engineering.md`:

> **Do not rely on VEO for readable text.** Signs, shirts, posters, storefronts, and UI elements render as plausible-looking gibberish. The existing "clean negative space for logos" rule from our image skill applies double for video - describe the area as "blank" or "out of frame" and composite text in post-production using a video editor.

### Object insertion - future v3.6.0 feature (affects ROADMAP.md only)

**Finding:** VEO 3.1 supports "object insertion" - adding elements into generated video while maintaining consistent lighting and physics. Our script doesn't support this.

**Change:** Add to v3.6.0 ROADMAP section alongside `--num-videos` and the parallel batch execution work. Not a v3.5.0 scope item.

### Rate limits (affects commit 4 docs)

**Finding:** 50 RPM for GA models, 10 RPM for preview endpoints, 10 concurrent operations per project, 4 videos per request max.

**Change for commit 4:** Add a "Rate limits" section to `veo-models.md`. Not a code change since our single-clip-at-a-time pattern is well within the limits, but worth documenting for users planning larger workloads.
